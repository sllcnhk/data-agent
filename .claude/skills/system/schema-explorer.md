# Schema 探索技能规程

## 核心职责

帮助用户**快速理解数据库表结构**，从业务角度解读字段含义，识别表间关联关系，
为后续查询、ETL 设计提供准确的元数据基础。

---

## 工作流程

### 第一步：获取表清单
1. 使用 `list_databases` 工具列出所有数据库
2. 使用 `list_tables` 工具列出目标数据库的所有表
3. 根据表名前缀推断数据分层（ODS/DWD/DWS/ADS）

**数据分层命名约定**：
| 前缀 | 层级 | 含义 |
|------|------|------|
| `ods_` | 原始层 | 直接来自业务系统的原始数据 |
| `dwd_` | 明细层 | 清洗、规范化后的明细数据 |
| `dws_` | 汇总层 | 按主题聚合的宽表/汇总表 |
| `ads_` | 应用层 | 直接服务业务报表的应用数据 |
| `dim_` | 维度层 | 业务维度表（用户、商品、地区等）|
| `fact_` | 事实层 | 业务事实表（订单、行为等） |

### 第二步：获取表详情
1. 使用 `describe_table` 工具获取字段列表和类型
2. 使用 `get_table_overview` 工具获取行数、分区等统计信息
3. 使用 `sample_table_data` 工具获取 3-5 行样本数据（帮助推断业务含义）

### 第三步：业务语义推断

根据字段名、样本数据和表名，推断字段的业务含义：

**常见字段模式识别**：
- `*_id` / `*_code` → 业务实体标识符
- `created_at` / `create_time` / `gmt_create` → 创建时间
- `updated_at` / `update_time` → 最后更新时间
- `is_deleted` / `del_flag` → 软删除标志（1=已删除）
- `status` / `state` → 状态码（需结合样本推断枚举含义）
- `user_id` / `uid` / `account_id` → 用户标识
- `amount` / `price` / `fee` → 金额类（注意单位：分 or 元）
- `cnt` / `count` / `num` → 数量类
- `dt` / `ds` / `date` → 日期分区字段

**输出格式示例**：
```
## 表：dwd_order_detail（订单明细表）

| 字段名 | 类型 | 业务含义 | 备注 |
|--------|------|----------|------|
| order_id | String | 订单ID | 主键，唯一标识一笔订单 |
| user_id | UInt64 | 用户ID | 关联 dim_user.user_id |
| amount | Decimal(18,2) | 订单金额 | 单位：元 |
| status | UInt8 | 订单状态 | 1=待付款 2=已付款 3=已完成 4=已取消 |
| dt | Date | 分区日期 | 按天分区，查询必须指定 |

**行数**: ~5,000万  **分区键**: dt
```

---

## 注意事项

1. **先采样再分析**：字段业务含义不明确时，必须采样 3-5 行数据辅助判断
2. **标注关键字段**：高危字段（金额、状态码）必须在业务含义中注明单位/枚举
3. **发现关联关系**：识别跨表 JOIN 可能性（`*_id` 字段关联）
4. **诚实的不确定性**：对推断不确定的字段，明确标注"推断：..."，建议用户确认
